# AGENTS.md

本文件是当前仓库的**代理协作开发规范**。任何进入本仓库进行分析、设计、编码、测试或文档补充的代理，都必须先阅读本文件，再阅读 `desc.md`、`codex-prompt.md`、`build.gradle.kts` 与现有源码。

---

## 1. 最高优先级约束

代理必须按以下优先级理解项目约束：

1. `desc.md`：产品范围、MVP 边界、API 范围、验收标准
2. `AGENTS.md`：多 agent 协作规则、开发规范、禁止事项、测试门槛
3. `codex-prompt.md`：面向自动编码代理的执行模板与分工建议
4. `build.gradle.kts`：技术栈与依赖现实边界
5. `src/main/resources/application.yaml`：现有配置基线
6. 当前仓库中已有源码、模板、测试与文档

如上述内容之间出现冲突，优先按更高优先级文件执行；如仍无法判断，选择**最小可运行实现**，并在文档中注明取舍。

---

## 2. 项目目标与范围

当前项目目标是实现一个**文件网盘系统 MVP**，技术方向已经固定，不允许随意偏离。

### 2.1 必做 MVP

- 基础登录能力
- 文件夹管理
- 文件上传 / 下载 / 删除 / 重命名 / 移动
- 文件列表与搜索
- 文件分享链接
- 图片 / 文本基础预览
- Thymeleaf 页面
- 基础测试
- README 与运行说明

### 2.2 暂不强制首版实现

以下功能只能预留扩展点，不要抢跑：

- 标签和分类
- 评论讨论
- 文件版本控制
- 回收站
- 复杂权限系统（RBAC / 多租户 / 协作）
- 云存储
- 自动备份恢复
- 文件加解密
- 高级转码或复杂预览
- 审计报表后台

---

## 3. 固定技术栈

代理必须遵守当前仓库实际技术栈：

- Kotlin
- Java 21
- Spring Boot 4.x
- Spring Web MVC
- Thymeleaf
- Spring Data JDBC
- MySQL
- 本地文件系统存储文件内容

### 3.1 明确限制

1. MVP 阶段**不要引入 React / Vue / Angular 独立前端工程**。
2. 前端页面优先使用 `Thymeleaf + 原生 JavaScript + CSS`。
3. 文件内容放本地存储，元数据放 MySQL。
4. 不要无故替换为 JPA、MongoDB、Redis、对象存储等其他实现。
5. 不要为了“看起来高级”而引入与 MVP 无关的大型依赖。

如确实需要补充依赖，必须说明用途，并尽量只补：

- Spring Security
- Validation
- Actuator
- 测试依赖

## 4. 推荐目录与分层约定

所有代理都应尽量遵循以下分层：

- `config`：配置类、安全、属性绑定、Bean 定义
- `controller`：页面路由与 REST API
- `service`：业务逻辑
- `repository`：Spring Data JDBC 仓储
- `domain`：实体、枚举、领域对象
- `dto`：请求响应对象、表单对象、视图模型
- `exception`：业务异常与统一异常处理
- `support` / `common`：公共工具、结果封装、常量
- `templates`：Thymeleaf 页面
- `static`：静态资源

### 4.1 命名原则

- Controller 仅处理请求协调，不承载复杂业务
- Service 负责核心业务流程
- Repository 只负责数据访问
- DTO 不要与 Entity 混用
- 页面模型与 API DTO 可以分离

## 5. 共享契约冻结规则

多 agent 并行前，必须先冻结以下共享契约：

- 核心实体字段
- 数据库表结构
- API 路径
- DTO 名称与语义
- 配置项命名
- 模板文件命名
- 统一错误响应格式

### 5.1 未冻结前禁止事项

在共享契约冻结前，代理不得：

- 自行新增与他人重复但语义不同的 DTO
- 擅自发明另一套 API 路由
- 擅自修改实体主字段语义
- 页面先写死尚未定稿的接口路径

### 5.2 契约变更规则

如果必须调整已冻结契约：

1. 先说明变更原因
2. 同步更新受影响的 controller / service / repository / 页面 / 测试
3. 更新 `README.md` 或相关说明文档
4. 合并前确保没有残留旧接口与旧字段引用

## 6. 多 Agent 分工建议

如果运行环境支持多 agent / 子代理 / 并行开发，建议按以下角色分工。

### Agent 0：主控代理

职责：

- 阅读仓库与核心文档
- 输出总体计划
- 冻结共享契约
- 安排并行边界
- 合并变更
- 跑全量测试并做最终验收

### Agent A：契约与骨架

职责：

- 定义包结构
- 定义 `domain` / `dto` / `config` 骨架
- 设计表结构和初始化脚本
- 统一异常结构与返回格式
- 确定页面路由与模板命名

### Agent B：认证与安全

职责：

- Spring Security 配置
- 登录 / 登出 / 当前用户
- 密码加密
- 默认管理员初始化
- 后台访问控制

### Agent C：文件元数据与存储

职责：

- `FileNode` 相关实体和仓储
- 本地文件存储实现
- 文件夹创建、文件上传下载、删除、重命名、移动
- 搜索能力

### Agent D：分享与预览

职责：

- `ShareLink` 相关实体与服务
- 分享创建、过期校验、提取码校验
- 分享下载
- 图片 / 文本 / PDF 基础预览

### Agent E：页面与交互

职责：

- 登录页、文件列表页、分享页
- 页面交互、表单、操作反馈
- 响应式布局
- 适配最终后端接口

### Agent F：测试与文档

职责：

- 集成测试、Web 测试、核心服务测试
- `README.md`
- 配置说明、运行说明、验收清单

## 7. 串行与并行规则

### 7.1 必须串行完成的阶段

#### 阶段 1：仓库审查与契约冻结

必须先完成：

- 目录结构确认
- 表结构确认
- 实体字段确认
- API 路径确认
- 配置项前缀确认
- 统一异常和统一返回结构确认

#### 阶段 2：基础依赖与公共配置

必须先完成：

- 依赖补齐
- 基础配置类
- 属性绑定类
- 公共异常处理
- 存储目录与默认配置基线

### 7.2 可以并行推进的阶段

当阶段 1 和阶段 2 完成后，可以并行推进：

- 认证与安全
- 文件元数据与存储
- 分享与预览
- 页面与交互

### 7.3 合并顺序建议

1. 契约与骨架先合并
2. 认证与文件能力优先合并
3. 分享与预览随后合并
4. 页面联调在接口稳定后进行
5. 测试与 README 最后统一补全并回归

## 8. 数据与存储约束

代理在实现时必须遵守：

1. 二进制文件内容**不要存数据库**。
2. MySQL 只保存文件元数据、用户数据、分享信息。
3. 本地文件存储路径必须做安全处理，防止路径穿越。
4. 删除文件夹前必须明确是否允许递归删除。
5. 重命名、移动文件时，要同时维护元数据与物理文件路径的一致性。
6. 上传时必须考虑文件名清洗、大小限制、类型限制。
7. 分享链接必须采用随机高熵 token，并支持过期策略。

建议至少落地以下表：

- `users`
- `file_nodes`
- `share_links`

如有扩展表，命名与语义要清晰。

## 9. 编码规范

### 9.1 一般要求

- 优先写完整可运行实现，不要只留接口空壳
- 保持 Kotlin 风格统一
- 只做与当前任务直接相关的最小变更
- 避免无意义重构与大规模格式化
- 不要改动与当前目标无关的公共 API

### 9.2 Web 与 API 约定

- REST API 路径保持一致、语义明确
- 页面路由与 API 路由分开组织
- 请求参数做基础校验
- 错误响应保持一致
- 页面出现错误时要给出用户可理解提示

### 9.3 安全要求

- 密码必须哈希存储
- 文件访问必须做权限判断
- 分享过期后必须不可访问
- 上传路径和文件名必须清洗
- 下载与预览尽量使用流式处理

## 10. 禁止事项

以下行为默认禁止：

1. 未经说明直接引入新的前端工程
2. 未经说明将数据访问切换为 JPA 或其他持久化框架
3. 未经说明引入云存储作为首版默认实现
4. 未经说明删除已有测试或绕过测试
5. 未经说明修改已冻结共享契约
6. 未经说明大规模重命名目录和包结构
7. 为了省事把密码明文存储或跳过权限校验
8. 在没有验证的情况下声称“已可运行”“已通过测试”

---

## 11. 测试要求

任何代理完成改动后，至少要补或验证与自己改动相关的测试。

### 11.1 最低测试覆盖建议

- 应用启动测试
- 认证流程测试
- 创建文件夹测试
- 文件上传 / 下载测试
- 重命名 / 移动 / 删除测试
- 分享链接创建与访问测试

### 11.2 测试原则

- 优先保证核心服务测试和 Web 层测试
- 关键流程应有集成测试
- 新增功能没有测试时，至少补最小可验证路径
- 合并前应重新跑受影响测试

## 12. 任务执行与汇报格式

代理在开始和结束每个阶段时，都应输出清晰进度。

### 12.1 开始阶段时

说明：

- 本阶段目标
- 预计修改哪些文件
- 是否依赖已冻结契约

### 12.2 完成阶段时

说明：

- 修改了哪些文件
- 完成了哪些能力
- 跑了哪些验证
- 还有哪些未闭环项

## 14. 面向代理的执行口令

任何进入本仓库的代理，都应按以下顺序行动：

1. 阅读 `AGENTS.md`
2. 阅读 `desc.md`
3. 阅读 `codex-prompt.md`
4. 审查当前代码结构与测试
5. 先输出任务拆解与并行策略
6. 先冻结契约，再进入并行开发
7. 每阶段结束后自检编译与测试

如果运行环境支持多 agent，请使用多 agent；如果不支持，请严格按本文件定义的角色边界串行模拟执行。
