# 通用执行提示词（Codex / Copilot / Claude，支持多 Agent 并行）

> 用法：将“可直接发送给自动编码代理的提示词”整段复制给你的编码代理。该模板默认适配本仓库的多 Agent 协作规则。

---

## 可直接发送给自动编码代理的提示词

你现在是本仓库的主执行代理，需要把当前项目实现为一个**可运行的文件网盘系统 MVP**。

请先按以下优先级阅读并遵守约束：

1. `desc.md`
2. `AGENTS.md`
3. `codex-prompt.md`
4. `build.gradle.kts`
5. `src/main/resources/application.yaml`
6. 当前仓库代码与测试

你的目标是直接产出可运行交付物，而不是泛泛建议：

- 后端代码
- 页面模板
- 配置
- 数据库初始化脚本
- 测试
- `README.md`

---

## 一、硬约束

1. 技术栈必须与仓库一致：
   - Kotlin
   - Java 21
   - Spring Boot 4.x
   - Spring Web MVC
   - Thymeleaf
   - Spring Data JDBC
   - MySQL
2. MVP 阶段不得引入 React / Vue / Angular 独立前端工程。
3. 文件内容存本地，元数据存 MySQL。
4. 代码按 `controller / service / repository / domain / dto / config` 分层。
5. 优先实现 MVP，Phase 2 功能仅留扩展点。
6. 每个阶段结束时必须自检：可编译、可运行关键路径、受影响测试通过。
7. 安全要求最低必须满足：密码哈希、路径穿越防护、权限校验、分享 token 高熵随机、过期链接不可访问。
8. 不明确处默认最小可运行实现，并在 README 标注取舍。

---

## 二、第一条回复格式（必须遵守）

在开始大规模编码前，你的第一条回复必须按以下结构：

### 1) 阶段任务清单

给出 6~10 条 checklist 任务（按实现顺序）。

### 2) 串并行策略

明确：

- 哪些阶段必须串行
- 哪些模块可并行
- 如何控制合并冲突

### 3) 首批改动文件清单

列出将新增/修改的关键文件路径。

完成上述三项后再开始编码。

---

## 三、多 Agent 分工（支持并行）

如果运行环境支持多 Agent，请按以下角色并行；如果不支持，请按同边界串行模拟。

### Agent 0（主控）

- 冻结共享契约（实体字段、表结构、API、DTO、配置项、模板命名）
- 安排并行边界
- 合并代码并跑全量测试

### Agent A（契约与骨架）

- 包结构、核心实体、DTO、异常、统一响应
- `schema.sql` / 初始化脚本
- API 与模板命名规范

### Agent B（认证与安全）

- 安全配置、登录/登出/当前用户
- 密码加密、后台访问控制
- 默认管理员初始化（如采用）

### Agent C（文件元数据与存储）

- `FileNode` 实体与仓储
- 本地存储实现
- 文件夹、上传、下载、删除、重命名、移动、搜索

### Agent D（分享与预览）

- `ShareLink` 实体与服务
- 分享创建/校验/下载
- 图片与文本基础预览（PDF 可选）

### Agent E（页面与交互）

- 登录页、文件页、分享页
- Thymeleaf 模板与基础交互
- 响应式布局

### Agent F（测试与文档）

- 服务测试、Web 测试、关键集成测试
- `README.md`
- 配置与运行说明

---

## 四、串行与并行边界

### 必须串行

1. 审查仓库并冻结共享契约
2. 补齐基础依赖与公共配置

### 可以并行

在上述两步完成后并行推进：

- 认证与安全
- 文件与存储
- 分享与预览
- 页面与交互

### 合并顺序

1. 契约骨架
2. 认证 + 文件核心能力
3. 分享与预览
4. 页面联调
5. 测试与 README 回归

---

## 五、MVP 边界

优先实现：

- 基础登录
- 文件夹管理
- 上传 / 下载 / 删除 / 重命名 / 移动
- 文件列表与搜索
- 分享链接
- 图片 / 文本基础预览
- Thymeleaf 页面
- 基础测试与 README

仅预留扩展点（不强行首版实现）：标签、评论、版本、回收站、RBAC、云存储、复杂审计、自动备份、文件加解密、高级转码。

---

## 六、编码与质量门槛

1. 只做当前任务相关的最小变更，避免无意义重构。
2. Controller 不承载复杂业务，业务逻辑放 Service。
3. DTO 与 Entity 分离，错误响应保持统一。
4. 上传/下载/预览尽量流式处理。
5. 每阶段结束输出：
   - 修改文件清单
   - 完成能力
   - 验证结果
   - 未闭环项

---

## 七、测试与验收

至少覆盖：

- 应用启动
- 认证流程
- 文件夹创建
- 上传/下载
- 重命名/移动/删除
- 分享创建与访问

最终验收项（逐条汇报）：

- [ ] 代码已写入仓库
- [ ] 项目可编译
- [ ] 相关测试通过
- [ ] 应用可启动
- [ ] 主要页面可访问
- [ ] 核心 API 可用
- [ ] README 已更新
- [ ] 高级能力已标注扩展点

---

## 八、执行启动口令

现在开始执行：

1. 先输出“阶段任务清单 + 串并行策略 + 首批改动文件”。
2. 再按多 Agent 边界推进实现。
3. 每完成一个阶段，执行自检并汇报。
